# HotReload

`hotreload` is a self-built Go CLI that watches a project directory, rebuilds the application when code changes, and restarts the server automatically. It is designed for local development workflows where manually stopping, rebuilding, and re-running a Go service becomes repetitive and slow.

This project was built as a backend engineering assignment and intentionally avoids existing hot-reload frameworks such as `air`, `realize`, or `reflex`. The only external watcher dependency used is `fsnotify`.

## What it does

Run the CLI with:

```bash
hotreload --root ./myproject --build "go build -o ./bin/server ./cmd/server" --exec "./bin/server"
```

After startup, the tool will:

1. watch the target directory recursively,
2. trigger the initial build immediately,
3. debounce noisy editor save events,
4. cancel stale builds if newer changes arrive,
5. stop the currently running server,
6. rebuild the latest version,
7. restart the server if the build succeeds.

## Features

- Recursive directory watching with dynamic folder detection
- Ignore rules for `.git`, `node_modules`, `vendor`, build folders, and temporary editor files
- Explicit filtering of build/output binaries to avoid rebuild loops
- 150ms debounce window to collapse rapid save bursts into one rebuild
- Build cancellation using `context.Context`
- Serialized rebuild pipeline so overlapping changes do not race
- Real-time build and server log streaming
- Graceful stop followed by force-kill fallback for stubborn processes
- Process-tree termination support on Unix and Windows
- Crash-loop backoff if the server exits immediately after startup
- Basic regression tests for watcher, debouncer, builder, and runner behavior


```bash
go install github.com/hotreload/cli@latest
```

> Replace the module path if you publish the project under a different repository.

## Usage

### Required flags

| Flag | Description |
|------|-------------|
| `--root` | Root directory to watch recursively |
| `--build` | Shell command used to build the project |
| `--exec` | Shell command used to start the built server |

### Example

```bash
hotreload \
	--root ./myproject \
	--build "go build -o ./bin/server ./cmd/server" \
	--exec "./bin/server"
```

## Architecture

The implementation follows a four-stage pipeline:

### 1. Watcher

The watcher package recursively registers directories with `fsnotify`, filters noisy paths and temporary files, detects newly created directories, removes deleted directories from the watch registry, and emits only relevant filesystem events.

### 2. Debouncer

The debouncer absorbs multiple filesystem events generated by a single editor save and emits one rebuild trigger after a short quiet period.

### 3. Builder

The builder executes the configured build command with `exec.CommandContext`, streams stdout/stderr directly, and distinguishes build cancellation from true build failure using typed errors.

### 4. Runner

The runner starts the target process, streams logs in real time, stops the old process before each rebuild, handles process trees, and applies crash-loop backoff if the server exits too quickly.

### Main orchestration

`main.go` wires the pipeline together. It performs the initial build, serializes rebuild requests, cancels stale builds, handles shutdown signals, and ensures only the latest project state is rebuilt.

## Testing

Run the full test suite with:

```bash
make test
```

The test target uses a workspace-local Go temp directory to avoid Windows application control issues when test binaries are executed from the system temp folder.

Current tests cover:

- debounce behavior,
- runner start/stop behavior,
- builder cancellation and failure typing,
- watcher ignore rules and dynamic directory handling.

## Logging

The tool uses `log/slog` for operational logging. Build output and server output are streamed directly to the terminal so errors are visible immediately.

## Known limitations

- Restart time depends on project build time, so the exact end-to-end latency varies by project.
- Very large directory trees may hit OS watch limits.
- The repository still requires manual submission steps outside the codebase, such as GitHub sharing and the Loom recording.
